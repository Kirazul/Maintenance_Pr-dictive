02. Problem

Failure prediction is hard because failures are rare.

A normal accuracy score can look excellent while the model misses the events maintenance teams actually care about.

96.6%

Records are normal machine behavior.

3.4%

Records are machine failures.

RiskAccuracy trap

Predicting “no failure” most of the time can still look accurate.

NeedCatch failures

The useful model must identify rare failure cases early.

ControlLimit false alarms

Too many alerts make the system hard to trust.

04. User result

The output is designed around one decision.

The user does not need to interpret every ML metric. The dashboard shows probability, risk band, and action first.

StableContinue monitoring

Risk is below the alert policy. No immediate maintenance action.

WarningSchedule inspection

The machine is moving into a risk zone and needs planned attention.

CriticalInspect immediately

Failure probability is high enough to require urgent action.

05. Model evidence

Gradient boosting gives the best operational balance.

The comparison focuses on balanced accuracy, precision, and recall instead of raw accuracy.

06. Threshold policy

The threshold controls the tradeoff.

A lower threshold catches more failures but creates more alerts. A higher threshold reduces alerts but can miss failures.
